cms4apps


1. Introduction

Cms4apps provides a very flexible solution to manage the content of dynamic web applications.


application integration minimized
Cms4apps can be integrated with any dynamic web application. It only shares templates and content with the application. The application's logic is not integrated in the cms but rather mocked with managed data. An Application with a shopping cart will store the contents of the cart in the session while cms4apps simply provides the author with an editor to create mock cart content. This works well for side-effect free template languages like handlebars.

template language agnostic
cms4apps can be integrated with different templating frameworks Handlebars is supported out-of-the-box.

datastore agnostic
Any rest server with crud and query operations can be used as a backend for content data. JCR, MongoDb and localstorage are supported out-of-the-box. Others can be integrated by implementing the javascript store abstraction.

powerful modeling
Cms4apps provides a sophisticated metadata editor to create the content author's user interface. No need to learn a complex dsl to create ui metadata.

Integration
Cms4apps is based on gform-app which makes it possible to provide data and content management for different dynamically or statically typed data stores side-by-side in a single tool.


2. Getting started

The application consists of three views which can be switched in the selector in the top toolbar.


page view

this view provides a tree and a grid to browse the pages in the left pane. It provides a context menu to delete pages and to edit pages in raw json form. The grid subview has a gui for creating simple queries and a text editor for creating more advanced queries. The gui queries only support the standard attributes name, parent and template. The text editor supports queries on all top level attributes. The format is mongodb-like. To search for page with a name that contains a digit use this query: {"name":{"$regex":"\\d"}}

In the center pane is the page preview. It displays the the rendered html of the currently selected page.

On the right there is the detail editor. Each page is opened in a new tab. The context menu in the tab header supports closing all or all the other tabs. This is where the author edits the content. On the top of the detail editor is a drop down list and button. The template of the page can be changed here. The Button switches to the detail editor of the currently selected template.

In the top toolbar there is a button to add a new page and to expand the size of the preview.

template view

this view provides a grid to browse the templates on the left and a detail editor on the right. The templates provide the following information

  • the ui that the author uses to edit the content
  • the actual handlebars template
  • the schema of the content including validation rules


partial view

The partial view is similar to the template view. The main difference between templates and partials is that a partial cannot be instantiated to create a new page. Partials can be used as components of templates.

file view

The file view provides access to raw files. These can be in text or binary format. Text format are identified by the file extensions and an appropriate ace editor is displayed. Among the supported formats are handlebars (*.hbs), css (*.css), markdown (*.md) and javascript (*.js). If the binary format is an image it will be displayed. Otherwise a download link is provided. Upload is possible via a file input field.



This cms4apps instance uses localstorage as data store. You may edit all of the content. Once you press reset store in the top toolbar all your data will be lost and the original will be restored. To export the current content open the console and call appController.exportToJson("page"). Repeat with the stores /template and partial.
